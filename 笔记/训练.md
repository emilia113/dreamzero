# DreamZero 训练笔记

## 1. 项目概述

**DreamZero** 是 NVIDIA GEAR Lab 开发的 **World Action Model (WAM)**，基于视频生成模型 **Wan2.1-I2V-14B-480P** 微调而来。核心思想是联合预测视频和动作，实现对未见机器人任务的零样本迁移。

- 论文名称: *World Action Models Are Zero-Shot Policies*
- 预训练模型: DreamZero-DROID (14B)、DreamZero-AgiBot
- 支持多具身体（DROID、AgiBot、YAM 等）

## 2. 模型架构

### 2.1 整体结构

```
VLA (Vision-Language-Action Model)
├── Backbone: IdentityBackbone（透传，不做处理）
└── Action Head: WANPolicyHead（核心策略模块）
    ├── Text Encoder (umt5-xxl) ── 冻结
    ├── Image Encoder (CLIP ViT-Huge-14) ── 冻结
    ├── VAE (Wan2.1 VideoVAE, z_dim=16) ── 冻结
    └── Diffusion Backbone: CausalWanModel ── 可训练
        ├── Patch Embedding (Conv3d: 36 → 5120)
        ├── Text Embedding Projection (4096 → 5120)
        ├── Time Embedding (sinusoidal + SiLU + Linear)
        ├── 40 层 Transformer Blocks
        │   ├── CausalWanSelfAttention（视频+动作+状态联合注意力）
        │   ├── Cross-Attention（语言条件）
        │   └── FFN (dim=5120, ffn_dim=13824)
        ├── Action Encoder (MultiEmbodimentActionEncoder) ── 可训练
        ├── State Encoder (CategorySpecificMLP) ── 可训练
        ├── Action Decoder (CategorySpecificMLP) ── 可训练
        └── Video Head (CausalHead) ── 可训练
```

## 3. 动作模块详解

动作模块并非一个独立的子网络，而是**深度嵌入在扩散骨干（CausalWanModel）中**。

### 3.1 MultiEmbodimentActonEncoder（动作编码器）

**职责**：把「带噪声的动作 + 扩散时间步」映射成 DiT 可以 attend 的 token，作为 DiT 的条件输入。

```
输入:
  noisy_actions  (B, T, action_dim)  ← 加噪后的动作（不是干净动作）
  timesteps      (B, T)              ← 当前扩散时间步
  cat_ids        (B,)                ← 具身体类别

流程:
  noisy_actions → CategorySpecificLinear(action_dim → hidden) → a_emb
  timesteps     → SinusoidalPositionalEncoding               → tau_emb
  concat([a_emb, tau_emb]) → Linear(2*hidden → hidden) → swish → Linear → x
  x → Linear(hidden → 5120)

输出: action_tokens (B, T, 5120)
```

- 使用 **CategorySpecificLinear**：每种具身体有独立的权重矩阵，支持多具身体联合训练
- 输入的是**带噪动作**而非干净动作，这是扩散模型的标准设计：编码器需要感知当前的噪声状态
- 时间步编码让模型知道「现在处于扩散过程的哪个阶段」

### 3.2 StateEncoder（状态编码器）

**职责**：把当前机器人状态映射成 DiT 的条件 token，作为去噪过程的固定条件。

```
输入:  state (B, T, state_dim)  ← 干净的状态观测（不加噪）
       cat_ids (B,)
输出:  state_tokens (B, T, 5120)
```

- 使用 CategorySpecificMLP（两层，中间 ReLU）
- 输入是**干净状态**，与扩散噪声无关，在整个去噪过程中保持不变

### 3.3 ActionDecoder（动作解码器）

**职责**：把 DiT 在动作 token 位置输出的隐状态解码回动作空间，得到模型的预测结果。

```
输入:  DiT 动作位置的隐状态 (B, T, 5120)
       cat_ids (B,)
输出:  action_pred (B, T, action_dim)
```

- 使用 CategorySpecificMLP
- **当前**输出的是**速度场预测**（velocity = ε - x0），而非干净动作
- 推理时，此输出送给 `FlowUniPCMultistepScheduler.step()` 来更新带噪动作

### 3.4 注意力机制中的动作处理（CausalWanSelfAttention）

```
序列结构（按 token 排列）:
[First Image (条件帧)] [Image Block 0..N] [Action Block 0..N] [State Block 0..N]

注意力流向:
- Image blocks: 关注 first image + 局部图像历史 + 当前 action/state
- Action blocks: 关注 image blocks + self + 当前 state
- State blocks: 仅自注意力（作为条件输入）
```

动作 token 与视频 token 在 40 层 Transformer 中做**联合注意力**，深度耦合。

## 4. 流程

### 4.1 训练流程 （单次前向，有 GT）

训练时对每个样本随机采样一个扩散时间步 t，做**一次前向传播**，不存在迭代去噪。

```
输入: 干净动作 x0 (B, T, action_dim)，干净状态 s，视频帧 v，文本 lang

Step 1: 采样噪声与时间步
  noise_action ~ N(0, I)
  t ~ Uniform(0, 1000)  [或 Beta 分布，取决于噪声策略]

Step 2: 加噪（Flow Matching 插值）
  noisy_actions = (1 - σ_t) * x0 + σ_t * noise_action

Step 3: 编码
  action_tokens = ActionEncoder(noisy_actions, t, embodiment_id)
  state_tokens  = StateEncoder(state, embodiment_id)
  [视频、文本同步编码]

Step 4: DiT 前向
  hidden = CausalWanModel(video_tokens, action_tokens, state_tokens, text_cond)

Step 5: 解码
  velocity_pred = ActionDecoder(hidden[action_positions])
  video_pred    = VideoHead(hidden[video_positions])

Step 6: 计算损失
  velocity_target = noise_action - x0          ← GT 速度场
  action_loss = MSE(velocity_pred, velocity_target) * scheduler_weight
  dynamics_loss = MSE(video_pred, video_velocity_target) * scheduler_weight
  loss = dynamics_loss + action_loss
```

噪声采样策略（三种，通过 config 切换）：

- **Standard**：视频和动作均匀采样时间步
- **High Noise Emphasis**：均使用 Beta 分布偏向高噪声
- **Decoupled**：视频用 Beta 分布，动作用独立均匀采样

### 4.2 推理流程（16 步迭代去噪）

推理时从纯噪声出发，迭代 16 步逐步去噪，每步调用一次 DiT。

```
初始化:
  noisy_input_action = randn(B, action_horizon, action_dim)  ← 纯高斯噪声
  [视频同步初始化]

for i in range(16):  # t: 1000 → 0

    Step 1: 编码（每步用当前噪声级别的动作）
      action_tokens = ActionEncoder(noisy_input_action, t_i, embodiment_id)
      state_tokens  = StateEncoder(state, embodiment_id)   ← 固定不变

    Step 2: DiT 前向
      hidden = CausalWanModel(...)

    Step 3: 解码得到速度场预测
      velocity_pred = ActionDecoder(hidden[action_positions])

    Step 4: 调度器更新（速度场积分，噪声减少一步）
      noisy_input_action = FlowUniPCMultistepScheduler.step(
          model_output=velocity_pred,
          timestep=t_i,
          sample=noisy_input_action,
      )

最终: noisy_input_action = 干净动作 x0  ✓
```

## 5. 微调策略

### 6.1 LoRA 微调（推荐）

- `train_architecture=lora`
- 目标模块: `q,k,v,o,ffn.0,ffn.2`（注意力层和 FFN）
- LoRA rank=4, alpha=4
- Text/Image Encoder 和 VAE 始终冻结，只训练扩散骨干中的 LoRA 适配器
- 显存占用低，适合单机多卡

### 6.2 仅微调动作相关子模块

可以手动冻结 Transformer 主体，只训练 ActionEncoder + ActionDecoder + StateEncoder：

```python
for param in model.parameters():
    param.requires_grad = False
for name, param in model.named_parameters():
    if any(k in name for k in ['action_encoder', 'action_decoder', 'state_encoder']):
        param.requires_grad = True
```

> 由于动作和视频是联合扩散的，完全隔离动作模块可能导致性能下降。

- `train_architecture=`
- 使用 DeepSpeed ZeRO-2 + CPU Offload
- 学习率建议 1e-5（比 LoRA 低一个量级）

## 我的方案：预测干净动作（x0 prediction）

### 核心思路

模型输出不再输出速度，而是输出干净动作。速度是$$z_t$$关于$t$的导数。在dreamzero的设定中。$z_t=(1-\sigma) * x_0 + \sigma * x_1$， 因此$v=(z_t - x_0) / \sigma$

模型输出和训练框架是**完全解耦**的：

- **训练框架**始终用速度场（velocity）做损失，这一层不需要改
- **模型内部**怎么得到 velocity 是模型自己的事

当前：ActionDecoder 直接输出 velocity → 拿去和 velocity GT 比
改后：ActionDecoder 输出 x0_pred（干净动作）→ 算 `velocity_pred = (noisy - x0_pred) / (σ + 0.01)` → 再拿去和 velocity GT 比

**关键区别**：

1. 不是改 training_target（GT 那侧），而是改模型输出到 loss 之间的路径（pred 那侧）
2. 不能用 `noise - x0_pred` 算 velocity，因为噪声对模型不可知（推理时没有噪声）
3. 正确公式从 `noisy = (1-σ)*x0 + σ*noise` 反推：`velocity = (noisy - x0) / σ`
4. 加 epsilon `(σ + 0.01)` 避免 σ 接近 0 时数值不稳定

### 代码框架回顾

```
训练时数据流（WANPolicyHead.forward）：

  ┌─ 训练框架侧（不改）──────────────────────────────────────────┐
  │                                                             │
  │  x0 (clean action)                                          │
  │   ├→ add_noise → noisy_action = (1-σ)*x0 + σ*noise         │
  │   └→ training_target_action = noise - x0  (velocity GT)     │
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  ┌─ 模型侧（要改）─────────────────────────────────────────────┐
  │                                                             │
  │  noisy_action → ActionEncoder → DiT (40层) → ActionDecoder  │
  │                                                             │
  │  当前：ActionDecoder 输出 velocity_pred                      │
  │  改后：ActionDecoder 输出 x0_pred                            │
  │        → velocity_pred = (noisy_action - x0_pred) / (σ+0.01)│
  │                                                             │
  └─────────────────────────────────────────────────────────────┘

  loss = MSE(velocity_pred, training_target_action)  ← 不变
```

### 训练侧改动（line ~794-799）

```python
# action_noise_pred 现在代表 x0_pred，需要先转成 velocity
sigma_action = self.scheduler.sigmas[timestep_action_id].to(device=self._device, dtype=action_noise_pred.dtype)
while len(sigma_action.shape) < len(action_noise_pred.shape):
    sigma_action = sigma_action.unsqueeze(-1)
action_velocity_pred = (noisy_actions - action_noise_pred) / (sigma_action + 0.01)

action_loss = MSE(action_velocity_pred, training_target_action)  # 后续 loss 代码不变
```

training_target_action（velocity GT = noise - x0）**不改**，scheduler 相关代码**不改**。

### 推理侧改动（line ~1307-1310）

action decoder 的输出 `flow_pred_cond_action` 现在代表 x0_pred（和训练一致，action decoder 在 `CausalWanModel._forward_inference` 内部已经调过了），传给 scheduler.step 之前需要转换：

```python
sigma_t = sample_scheduler_action.sigmas[index]
action_velocity = (noisy_input_action - flow_pred_cond_action) / (sigma_t + 0.01)
noisy_input_action = sample_scheduler_action.step(
    model_output=action_velocity, ...)
```

### 各模块的处理方式


| 模块                          | 是否需要改    | 说明                                |
| --------------------------- | -------- | --------------------------------- |
| ActionEncoder               | 不改       | 输入仍是带噪动作 + 时间步                    |
| StateEncoder                | 不改       | 与预测目标无关                           |
| DiT Transformer blocks      | 不改       | 隐状态的含义由 decoder 和 loss 共同决定       |
| ActionDecoder (MLP)         | **不改结构** | 同样的 MLP，只是学到的映射含义从 velocity 变成 x0 |
| VideoHead                   | 不改       | 视频仍然预测 velocity                   |
| FlowMatchScheduler          | 不改       | 训练用，training_target 仍然是 velocity  |
| FlowUniPCMultistepScheduler | 不改       | 推理用，仍然接收 velocity 输入              |


**实际需要改代码的只有 `wan_flow_matching_action_tf.py` 中两处**：训练 loss 前的转换、推理 step 前的转换。
