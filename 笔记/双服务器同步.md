## 日常同步流程

### 场景 A：主开发机 → 中电服务器

**主开发机上：**

```bash
git add <修改的文件>
git commit -m "你的提交信息"
git push origin main
```

**中电服务器上：**

```bash
# 方案1：本地没有未提交改动
git pull origin main --ff-only

# 方案2：本地有未提交改动
git stash
git pull origin main --ff-only
git stash pop
```

### 场景 B：中电服务器 → 主开发机

反过来同理，在中电服务器 commit + push，在主开发机 pull。

---

## 避免冲突的关键原则

> **同一时间只在一台服务器上改代码，改完 push，另一台再 pull。**

冲突的根源是两台机器同时修改了同一个文件。如果遵守"改完就推，拉完再改"，就永远不会冲突。

## 出问题了怎么办

### 问题1：pull 时报 "divergent branches"

说明两边都有对方没有的 commit。

**如果本地改动不重要（最常见）：** 直接丢弃本地 commit，对齐远程：

```bash
git stash                       # 保存未提交的改动
git fetch origin main
git reset --hard origin/main    # 本地 commit 历史强制对齐远程
git stash pop                   # 恢复未提交的改动
```

**如果本地改动要保留：** 用 rebase 把本地 commit 接到远程后面：

```bash
git stash
git pull --rebase origin main
git stash pop
```

### 问题2：stash pop 时文件冲突

```bash
# 查看哪些文件冲突
git status

# 选择保留哪个版本：
# 保留本地 stash 的版本（中电服务器自己改的）
git checkout --theirs <冲突文件>

# 保留远程版本（从主开发机推过来的）
git checkout --ours <冲突文件>

# 标记解决 + 清理 stash
git add <冲突文件>
git stash drop
```

> `--ours` / `--theirs` 在 stash pop 场景下：
> - `--ours` = 当前分支（即刚 pull 下来的远程版本）
> - `--theirs` = stash 里的内容（本地改动）

### 问题3：不想处理冲突，想从头来

```bash
# 放弃所有未提交的改动，完全对齐远程
git checkout -- .               # 丢弃工作区改动
git fetch origin main
git reset --hard origin/main
# stash 里的内容还在，可以选择性恢复或丢弃
git stash drop                  # 丢弃 stash
```

## 懒人方案（核弹级）

如果 git 状态彻底乱了，不想修了：

```bash
# 备份你需要保留的本地文件
cp scripts/train/droid_training_lora.sh ~/backup_lora.sh

# 强制对齐远程
git fetch origin main
git reset --hard origin/main
git clean -fd                   # 删除所有未跟踪文件（慎用）

# 把备份的文件复制回来
cp ~/backup_lora.sh scripts/train/droid_training_lora.sh
```

## 首次配置（中电服务器，只需一次）

```bash
git config user.name "yyu"
git config user.email "yyu@example.com"
git config pull.ff only    # pull 默认只允许 fast-forward，避免自动产生 merge commit
```

`pull.ff only` 的好处：如果有 divergent branches，git 会直接报错而不是自动 merge，让你主动选择怎么处理。
